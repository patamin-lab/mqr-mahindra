# Google Drive Architecture

Google Drive is the platform's media repository (`docs/TECH_STACK.md`): photos, videos, and generated PDFs are stored here and referenced from Supabase records by URL/ID — never stored as binary data in the database (`docs/ARCHITECTURE_PRINCIPLES.md`).

## Current State

A working OAuth2-based integration already exists: `src/lib/googleDrive.ts` and the credential-setup script `scripts/get-google-refresh-token.mjs` (`package.json` script: `drive:auth`). This document defines the target folder structure and conventions that integration should converge on as it's extended to serve modules beyond MQR — it does not describe a from-scratch build.

## Folder Structure

```
/MSEAL SERVICE SYSTEM/
  /<module-name>/                e.g. /MQR/, /PM Record/, /Warranty/
    /<entity-type>/               e.g. /Records/, /Claims/, /Deliveries/
      /<record-id>/
        /photos/
        /videos/
        /pdf/
```

- Top-level folder per business module, matching the module names in `docs/VISION.md` (kebab-free, human-readable display names are fine in Drive since Drive folder names are not code identifiers).
- One sub-folder per record, so all media for a single record is easy to locate and permission independently if ever needed.
- Media type sub-folders (`photos/`, `videos/`, `pdf/`) keep each record folder browsable instead of a flat dump of mixed files.

## Naming Convention

Files: `<record-id>_<category>_<timestamp>.<ext>`

- `record-id` — the Supabase record's identifier, so a file can always be traced back to its source row.
- `category` — what the file is (e.g. `before`, `after`, `signature`, `claim-form`).
- `timestamp` — capture/generation time, to disambiguate multiple files of the same category and preserve chronological order.

This avoids storing ambiguous original filenames straight from a phone camera (e.g. `IMG_2031.jpg`), consistent with `docs/NAMING_STANDARD.md`.

## Photo Storage

- Photos are normalized on upload (including HEIC → standard format conversion, already available via the `heic-convert` dependency) before being stored, so downstream consumers (PDF generation, web display) don't need to handle every source format.
- Stored under the record's `photos/` sub-folder per the structure above.

## Video Storage

- Stored under the record's `videos/` sub-folder.
- Given dealer connectivity constraints (`docs/PRODUCT_PHILOSOPHY.md` — Fast Performance), video upload should support resumable/chunked upload where the underlying Drive API allows it, rather than requiring a single uninterrupted upload.

## PDF Storage

- PDFs generated by the `pdf` platform service (`docs/PLATFORM_SERVICES.md`) are stored under the record's `pdf/` sub-folder.
- A regenerated PDF (e.g. after a record correction) is stored as a new timestamped file rather than overwriting the prior version, preserving history.

## Permissions

- Folders are created and owned by the platform's service account/OAuth identity, not by an individual user's personal Drive.
- Sharing is link-based and scoped to "Anyone with the link" only where the content is non-sensitive and read-only access is intended (mirroring the existing pattern already used for the read-only reference sheet in `src/lib/tractorSheet.ts`); sensitive media follows least-privilege sharing instead.
- End users never need their own Drive account or permissions to use the platform — Drive is an implementation detail behind the `google-drive` service, not something a dealer interacts with directly.

## Retention

- Media is retained for as long as its associated Supabase record exists. Deleting a record's media (if ever required) is a deliberate, explicit action — not an automatic side effect of any other process, consistent with the Git/data safety posture this platform follows elsewhere.
- Long-term archival/cold-storage policy (e.g. moving very old media to a cheaper tier) is a future operational decision, not defined by this sprint.
